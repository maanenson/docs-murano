<h1 id="lua-scripting-api">Lua Scripting API</h1> <p>Exosite&#39;s One Platform can run Lua scripts on your behalf. These scripts have a rich set of capabilities and may be used to offload processing from your device.</p> <p>If you&#39;re completely new to Exosite&#39;s APIs, you may want to read the <a href="../README.md">API overview</a> first.</p> <h2 id="overview">Overview</h2> <p>Scripts are written in Lua 5.2 scripting language. For general information about Lua 5.2, please reference the <a href="http://www.lua.org/manual/5.2/">online Lua manual</a>.</p> <p>Scripts may be added to an application either using Portals, or by using the <a href="https://github.com/exosite/docs/tree/master/rpc">Remote Procedure Call API</a>. In Portals, scripts are added on the <a href="https://portals.exosite.com/manage/scripts">script management page</a>.</p> <p>To use the RPC to create a script, use the <a href="https://github.com/exosite/docs/tree/master/rpc#create-datarule">create</a> RPC procedures. It&#39;s also possible to use the <a href="https://github.com/dweaver/exoline">Exoline</a> utility to upload a script from the command line using the <code>script</code> Exoline command. Run this command for details:</p> <pre><code>$ exo script --help </code></pre><p>Scripts are associated with a client and may read, write, record and subscribe to data for any resources that client owns. For example, in Portals, if a script A is added to a particular device client it may access all of that device&#39;s datasources, but not the datasources of any other devices. If script B is added to a portal, it will have access to portal-level datasources as well as devices in that portal and their datasources. </p> <p>A script may also dispatch messages using transports such as HTTP, SMS, Email, and XMPP.</p> <h2 id="examples">Examples</h2> <p>Examples of One Platform Lua scripts are made available in this repository: <a href="https://github.com/exosite-garage/PlatformScripts">Example Platform Scripts Repository</a></p> <h2 id="the-script-environment">The script environment</h2> <p>Scripts are completely isolated from one another, each running in its own, secure environment, complete with access to a set of Lua tables and functions, as well as a set of One Platform API functions, the &#39;alias&#39; table and Global properties and functions.</p> <p>Scripts are scheduled to run with a limited number of execution ticks. When these ticks are consumed, the script is suspended until subsequently rescheduled.</p> <p>Scripts also have memory usage limits, which if completely consumed terminates the script with a &#39;not enough memory&#39; error. Please reference the Lua 5.2 manual for how Lua manages memory. In particular:</p> <ul> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#2.5">Garbage Collection</a></p> </li> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#3.5">Visibility Rules</a></p> </li> </ul> <h3 id="lua-tables-and-functions">Lua tables and functions</h3> <p>The following global Lua tables and functions are available to Lua scripts. They operate exactly as described in the Lua 5.2 reference manual.</p> <ul> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#6.7"><code>bit32</code></a></p> </li> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#6.6"><code>math</code></a> (Note: the <code>math.randomseed</code> function is not available to scripts.)</p> </li> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#6.4"><code>string</code></a> (Note: the <code>string.dump</code> and <code>string.gsub</code> functions are not available to scripts.)</p> </li> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#6.5"><code>table</code></a> (Note: the <code>table.sort</code> function is not available to scripts.)</p> </li> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#pdf-ipairs"><code>ipairs()</code></a></p> </li> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#pdf-next"><code>next()</code></a></p> </li> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#pdf-pairs"><code>pairs()</code></a></p> </li> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#pdf-select"><code>select()</code></a></p> </li> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#pdf-tonumber"><code>tonumber()</code></a></p> </li> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#pdf-tostring"><code>tostring()</code></a></p> </li> <li><p><a href="http://www.lua.org/manual/5.2/manual.html#pdf-type"><code>type()</code></a></p> </li> </ul> <h3 id="one-platform-tables">One Platform tables</h3> <p>The following One Platform resources and features are available to Lua scripts:</p> <ul> <li><a href="#the-alias-table"><code>alias</code></a></li> <li><a href="#the-dispatch-table"><code>dispatch</code></a></li> <li><a href="#the-manage-table"><code>manage</code></a></li> </ul> <h3 id="global-tables-functions-and-properties">Global tables, functions, and properties</h3> <p>The following global resources and features are available to Lua scripts:</p> <ul> <li><a href="#the-json-global-table"><code>json</code></a></li> <li><a href="#utilities"><code>date()</code></a></li> <li><a href="#utilities"><code>debug()</code></a> </li> <li><a href="#utilities"><code>setlocale()</code></a></li> <li><a href="#utilities"><code>settimezone()</code></a></li> <li><code>now</code> holds the current UNIX timestamp. Note that <code>now</code> causes a One Platform API call behind the scenes so that it stays in step with the One Platform time. For best performance we recommend calling this at most once per iteration of a looping script. For example, if three different variables are to be initialized to the current timestamp in a row, <code>now</code> should be called once, its value stored in a local variable and subsequently used to initialize the variables.</li> </ul> <h2 id="the-alias-table">The alias table</h2> <p>Every script is owned by a One Platform client, and has access to that client and all of its resources through the <code>alias</code> table. </p> <ul> <li>The client: <code>alias[&#39;&#39;]</code></li> <li>The client&#39;s owned resource (by alias): <code>alias[&#39;myAlias&#39;]</code></li> <li>The client&#39;s owned resource (by resource ID): <code>alias[{rid = &#39;64c217b5243eeda29d4101234567890123456789&#39;}]</code></li> </ul> <p>It is a good idea to create a local reference to specific aliases used in the script, e.g.:</p> <pre><code>local room_temp = alias[&#39;room_temp&#39;] </code></pre><p>Each alias has properties and functions through which the script can interact with the aliased resource (for example to read from a dataport or write to a dataport)</p> <ul> <li><p><code>.alias</code> </p> <p> This sub alias table is only available if the referenced alias is a client. It provides the same functionality as the global alias table but for the referenced client.</p> </li> <li><p><code>.manage</code></p> <p> The manage table is only available if the referenced alias is a client. For the functionality provided, see <a href="#the-manage-table">The manage table</a>.</p> </li> <li><p><code>.meta</code> </p> <p> This provides read-only access to the <code>meta</code> field of the resource.</p> </li> <li><p><code>.name</code> </p> <p> This provides read-only access to the <code>name</code> field of the resource.</p> </li> <li><p><code>.status</code> </p> <p> Indicates the outcome of the most recent activity performed on the Alias. Actions include the reading or writing of any of the Alias properties and functions, excluding <code>.last</code> and <code>.status</code> itself.</p> <p> The actual values this property can return are listed with each property and function that can affect the value of <code>.status</code>. This property is read only.</p> </li> <li><p><code>.timestamp</code></p> <p> The timestamp of the most recent datapoint written to the resource. If the resource contains no datapoints, <code>.timestamp</code> will read <code>nil</code>.</p> </li> </ul> <pre><code> ------------------------------------------------------------------------------ Values: timestamp :: number | nil ------------------------------------------------------------------------------ Status: &#39;ok&#39; :: string - accessing .timestamp was successful </code></pre><ul> <li><p><code>.value</code></p> <p> When read, it returns the value of the most recent datapoint written to the resource. If the resource contains no datapoints, <code>.value</code> will read <code>nil</code>. When written, the given value gets written to the resource with the current timestamp.</p> </li> </ul> <pre><code> ------------------------------------------------------------------------------ Values: value :: boolean | number | string | nil ------------------------------------------------------------------------------ Status: &#39;ok&#39; :: string - accessing .value was successful </code></pre><ul> <li><p><code>.last</code></p> <p> When read, it returns the timestamp returned by the most recent call to <code>.wait</code> function. When written, the current time, as the script perceives it, will be changed to the given timestamp. Thus it is possible to set <code>.last</code> to a point in the past and iterate through historical data using <code>.wait</code>. When the script starts <code>.last</code> is initialized to the value of <code>.timestamp</code>.</p> </li> </ul> <pre><code> ------------------------------------------------------------------------------ Values: last :: number </code></pre><ul> <li><p><code>.wait([expire])</code></p> <p> Returns the timestamp for the next unprocessed datapoint. Next being relative to the timestamp held by <code>.last</code> property. If there is no datapoint available, the call will block. The <code>expire</code> parameter sets the wait expiration timestamp in case no datapoint becomes available. When the call returns, <code>.last</code> property will be updated to the timestamp of the next unprocessed datapoint or the wait expiration timestamp.</p> <p> Note that the <code>expire</code> period is non deterministic. Although the One Platform honors the <code>expire</code> parameter, there may be small fluctuations of up to a few seconds beyond what is specified by the <code>expire</code> parameter.</p> </li> </ul> <pre><code> ------------------------------------------------------------------------------ Arguments: :: nil - no expiration expire :: number - expiration timestamp ------------------------------------------------------------------------------ Returns: timestamp :: number - timestamp of the next unprocessed datapoint :: nil - wait expired ------------------------------------------------------------------------------ Status: &#39;ok&#39; :: string - the call has returned without expiring &#39;expired&#39; :: string - the call has expired </code></pre><ul> <li><p><code>[&lt;timestamp&gt;]</code></p> <p> Indexing an alias with a numerical timestamp makes it possible to read and record historical data.</p> <p> When read, the value of the datapoint with the given timestamp. If no value exists for the given timestamp then <code>nil</code> will be returned.</p> <p> When written, the given non-nil value gets recorded to the resource at the given timestamp. When nil is specified, the datapoint at the given timestamp will be deleted.</p> </li> </ul> <pre><code> ------------------------------------------------------------------------------ Values: [&lt;timestamp&gt;] :: boolean | number | string | nil ------------------------------------------------------------------------------ Status: &#39;ok&#39; :: string - accessing [&lt;timestamp&gt;] was successful </code></pre><h2 id="the-manage-table">The manage table</h2> <pre><code>The manage table provides resource management functionality.
