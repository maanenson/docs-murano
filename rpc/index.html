<h1 id="remote-procedure-call-api">Remote Procedure Call API</h1> <p>The JSON RPC API provides full featured access to data and resources on the One Platform. It is intended for applications that need to do more than just read and write individual points.</p> <p>If you&#39;re completely new to Exosite&#39;s APIs, you may want to read the <a href="../README.md">API overview</a> first.</p> <h4 id="time-series-data">Time Series Data</h4> <p><a href="#read">read</a> - read time series data</p> <p><a href="#write">write</a> - write new data into a time series</p> <p><a href="#writegroup">writegroup</a> - write new data into multiple time series</p> <p><a href="#record">record</a> - write time series data that has already been timestamped</p> <p><a href="#recordbatch">recordbatch</a> - write multiple time series data at specific timestamps</p> <p><a href="#flush">flush</a> - remove time series data</p> <p><a href="#wait">wait</a> - long polling, i.e., wait a data and get response when the data is updated</p> <h4 id="resources">Resources</h4> <p><a href="#create-client">create (client)</a> - create a resource that can contain other resources</p> <p><a href="#create-dataport">create (dataport)</a> - create a time series data resource</p> <p><a href="#create-datarule">create (datarule)</a> - create a resource that can perform processing in response to data</p> <p><a href="#create-dispatch">create (dispatch)</a> - create a resource for sending notifications </p> <p><a href="#create-clone">create (clone)</a> - copy an existing resource</p> <p><a href="#move">move</a> - move an existing resource</p> <p><a href="#update">update</a> - update an existing resource</p> <p><a href="#info">info</a> - get information about an existing resource</p> <p><a href="#listing">listing</a> - list the children of a client resource</p> <p><a href="#drop">drop</a> - delete a resource</p> <p><a href="#usage">usage</a> - get usage information for a resource</p> <h4 id="aliases">Aliases</h4> <p><a href="#map">map</a> - create an alias that can be used to refer to a resource</p> <p><a href="#lookup">lookup</a> - look up the resource for an alias</p> <p><a href="#unmap">unmap</a> - remove an alias for a resource</p> <h4 id="shares-and-keys">Shares and Keys</h4> <p><a href="#share">share</a> - generate a code that allows non-owners to access resources</p> <p><a href="#revoke">revoke</a> - revoke a share code or CIK</p> <p><a href="#lookup">lookup</a> - look up a resource based on a share code</p> <p><a href="#activate">activate</a> - activate a share code or CIK</p> <p><a href="#deactivate">deactivate</a> - deactivate a share code or CIK</p> <h2 id="api-libraries">API Libraries</h2> <p>Wrapper libraries are available for this API:</p> <ul> <li>Python: <a href="https://github.com/exosite-labs/pyonep">pyonep</a></li> <li>Java: <a href="https://github.com/exosite-labs/javaonep">javaonep</a></li> <li>C++: <a href="https://github.com/exosite-labs/cpponep">cpponep</a></li> <li>.NET: <a href="https://github.com/exosite-labs/clronep">clronep</a></li> <li>Cocoa: <a href="https://github.com/exosite-labs/cocoaonep">cocoaonep</a></li> </ul> <h2 id="conventions">Conventions</h2> <p>This document uses a few notational conventions:</p> <ul> <li>JSON is pretty printed for clarity. The extra whitespace is not included in the RPC JSON.</li> <li>Comments (<code>//</code>) are occasionally included in JSON to give hints or provide detail. These comments are not included in actual requests or responses.</li> <li>A name in angle brackets, e.g. <code>&lt;myvar&gt;</code>, is a placeholder that will be defined elsewhere.</li> <li><p><code>&lt;ResourceID&gt;</code> is a placeholder that may be either a 40 digit resource identifier (e.g., <code>&quot;879542b837bfac5beee201234567890123456789&quot;</code>) or an alias reference (e.g., <code>{&quot;alias&quot;: &quot;myalias&quot;}</code>). It may also be a self reference: <code>{&quot;alias&quot;: &quot;&quot;}</code>. See <a href="#identifying-resources">Identifying Resources</a> for details.</p> </li> <li><p><code>number</code> indicates a number, e.g. 42</p> </li> <li><code>string</code> represents a string, e.g. &quot;MySensor&quot;</li> <li><code>|</code> represents multiple choice</li> <li><code>=</code> represents default value</li> <li><code>...</code> represents one or more of the previous item</li> </ul> <h2 id="http-request-response-example">HTTP Request/Response Example</h2> <p>JSON RPC are HTTP POST requests with a body containing a JSON-encoded call. Here is a full example of an HTTP request, with JSON formatted for readability:</p> <pre><code>POST /onep:v1/rpc/process HTTP/1.1 Host: m2.exosite.com:80 Content-Type: application/json; charset=utf-8 User-Agent: API Example (danweaver@exosite.com) Content-Length: 235 { &quot;auth&quot;: { &quot;cik&quot;: &quot;5de0cfcf7b5bed2ea7a801234567890123456789&quot; }, &quot;calls&quot;: [ { &quot;id&quot;: 56, &quot;procedure&quot;: &quot;read&quot;, &quot;arguments&quot;: [ { &quot;alias&quot;: &quot;temperature&quot; }, { &quot;endtime&quot;: 1376957311, &quot;limit&quot;: 3, &quot;selection&quot;: &quot;all&quot;, &quot;sort&quot;: &quot;desc&quot;, &quot;starttime&quot;: 1 } ] } ] } </code></pre><p>Here is the full HTTP response for that request:</p> <pre><code>HTTP/1.1 200 OK Date: Tue, 20 Aug 2013 00:08:27 GMT Content-Length: 90 Content-Type: application/json; charset=utf-8 Connection: keep-alive Server: nginx [ { &quot;id&quot;: 56, &quot;result&quot;: [ [ 1376957195, 72.2 ], [ 1376957184, 72.3 ], [ 1376951473, 72.5 ] ], &quot;status&quot;: &quot;ok&quot; } ] </code></pre><h2 id="making-a-request">Making a Request</h2> <p>RPC API requests are HTTP POSTs to <code>/onep:v1/rpc/process</code> and may contain the following headers.</p> <table> <thead> <tr> <th>Header</th> <th>Description</th> <th style="text-align:center">Required?</th> </tr> </thead> <tbody> <tr> <td><code>Host: m2.exosite.com:80</code></td> <td>The host should be either <code>m2.exosite.com</code> for normal Portals accounts or <code>&lt;your domain&gt;.m2.exosite.com</code> for whitebox accounts. At the moment <code>m2.exosite.com</code> works for all types of accounts, but this is not guaranteed to be supported in the future. Both HTTP and HTTPS are supported.</td> <td style="text-align:center">Yes</td> </tr> <tr> <td><code>Content-Type: application/json; charset=utf-8</code></td> <td>Describes the content type of the body. For the JSON RPC, this is always <code>application/json; charset=utf-8</code></td> <td style="text-align:center">Yes</td> </tr> <tr> <td><code>User-Agent: API Example (danweaver@exosite.com)</code></td> <td>If possible your application should identify itself by putting contact information in the <code>User-Agent</code> header. This also is not enforced, but will help us with any support requests you have.</td> <td style="text-align:center">No</td> </tr> <tr> <td><code>Content-Length: 235</code></td> <td><code>Content-Length</code> contains the number of bytes in the body of the request.</td> <td style="text-align:center">Yes</td> </tr> </tbody> </table> <p>The body of a request must be valid JSON. See <a href="http://www.json.org">http://www.json.org</a> for details on the JSON format.</p> <p>The maximum number of consecutive HTTP requests that can be sent via a single connection to the One Platform is 100. In its response to the 100th request on the same connection, the One Platform will send the header &quot;Connection: close&quot; and then close the underlying TCP connection. The client should in this case create a new connection to the One Platform to continue.</p> <h2 id="authentication">Authentication</h2> <p>Requests to the JSON RPC are made on behalf of a particular client in the system, called the &quot;calling client&quot;. Every request passes a client key, called a CIK, that functions like a password and grants limited control over that client and full control of that client&#39;s subhierarchy. This is somewhat different from other APIs where authentication is done on behalf of a user account that is granted access to a set of resources.</p> <p>For example, to <a href="#read">read</a> from a Portals datasource, you could authenticate with the CIK of the device that owns the datasource or with the CIK of the portal that owns that device, all the way up to the root node of the One Platform tree.</p> <p>Authentication information is placed in the JSON body of a request, in <code>&quot;auth&quot;</code>. The value of the <code>&quot;auth&quot;</code> key can take any of these forms:</p> <ul> <li><p><code>{&quot;cik&quot;: CIK}</code> authenticates as the client identified by the given CIK. This is the most common form.</p> </li> <li><p><code>{&quot;cik&quot;: CIK, &quot;client_id&quot;: RID}</code> authenticates as the given client if the CIK identifies an ancestor of the given client.</p> </li> <li><p><code>{&quot;cik&quot;: CIK, &quot;resource_id&quot;: RID}</code> authenticate as the owner of the given resource if the CIK identifies as an ancestor of the given resource.</p> </li> </ul> <h2 id="identifying-resources">Identifying Resources</h2> <p>Many procedures in the API include an argument for identifying a resource to act upon. In this documentation, that resource is identified by <code>&lt;ResourceID&gt;</code>. This argument can take any of these forms:</p> <ul> <li><p><code>&quot;34eaae237988167d90bfc2ffeb666daaaaaaaaaa&quot;</code> directly identifies a resource ID. The resource must be in the <a href="#authentication">calling client</a>&#39;s subhierarchy.</p> </li> <li><p><code>{&quot;alias&quot;: &quot;temperature&quot;}</code> looks up an immediate child of the <a href="#authentication">calling client</a> by alias (<code>&quot;temperature&quot;</code> in this example).</p> </li> <li><p><code>{&quot;alias&quot;: &quot;&quot;}</code> identifies the calling client itself. So, for example, if <code>&quot;auth&quot;</code> was set to <code>{&quot;cik&quot;:&quot;e469e336ff9c8ed9176bc05ed7fa40daaaaaaaaa&quot;</code>, the procedure would act upon the client whose CIK is <code>&quot;e469e336ff9c8ed9176bc05ed7fa40daaaaaaaaa&quot;</code>.</p> </li> </ul> <h2 id="request-json">Request JSON</h2> <p>The body of a request has the following structure:</p> <pre><code>{ &quot;auth&quot;: { &quot;cik&quot;: &quot;e469e336ff9c8ed9176bc05ed7fa40daaaaaaaaa&quot; }, &quot;calls&quot;: [ // first call is a read { &quot;id&quot;: 1, &quot;procedure&quot;: &quot;read&quot;, &quot;arguments&quot;: [ // &lt;ResourceID&gt; // descendant RID of calling client &quot;34eaae237988167d90bfc2ffeb666daaaaaaaaaa&quot;, { &quot;starttime&quot;:1376709504, &quot;endtime&quot;:1376709527, &quot;limit&quot;:1, &quot;sort&quot;:&quot;desc&quot;, &quot;selection&quot;:&quot;all&quot; } ] }, // second call is a write { &quot;id&quot;: 2, &quot;procedure&quot;: &quot;write&quot;, &quot;arguments&quot;: [ // &lt;ResourceID&gt; // instead of an RID string, this call references the // alias of one of the calling client&#39;s children. { &quot;alias&quot;: &quot;temperature&quot; }, 65.4 ] } ] } </code></pre><ul> <li><code>&quot;auth&quot;</code> provides the authentication for the procedures listed in <code>&quot;calls&quot;</code>. See <a href="#authenticating">Authenticating</a> for details.</li> </ul> <p><code>&quot;id&quot;</code> is an identifier for the call, and may be a number or a string of up to 40 characters. A matching ID is returned in the response. If <code>&quot;id&quot;</code> is omitted for a particular call, no response will be returned. If <code>&quot;id&quot;</code> is omitted for all calls in <code>&quot;calls&quot;</code>, no response will be given for the entire request message.</p> <p><code>&quot;procedure&quot;</code> and <code>&quot;arguments&quot;</code> are specific to individual procedures and are documented <a href="#procedures-1">below</a>. </p> <h2 id="response-json">Response JSON</h2> <p>The response body is always JSON, but its format varies based on error handling conditions. If a call succeeds, the body of the response is a JSON list of responses to the calls made in the request:</p> <pre><code>[ // response for { &quot;id&quot;: 1, &quot;status&quot;: &quot;ok&quot;, &quot;result&quot;: [[1376709527, 64.2]] } ] </code></pre><ul> <li><code>&quot;id&quot;</code> identifies the corresponding request call. </li> <li><code>&quot;result&quot;</code> is the return value for the procedure. Procedures without return values omit it entirely.</li> </ul> <p>If a particular call fails, the response body is still a list, but <code>&quot;status&quot;</code> for the response for that call is set to something besides <code>&quot;ok&quot;</code>, and an <code>&quot;error&quot;</code> key is included:</p> <pre><code>[{&quot;id&quot;: 0, &quot;status&quot;: &quot;fail&quot;, &quot;error&quot;: {&quot;code&quot;: 501, &quot;message&quot;: &quot;Error&quot;, &quot;context&quot;: &quot;arguments&quot;}] </code></pre><p>If the request message causes an error not associated with any given call, the response body is a JSON object like this:</p> <pre><code>{&quot;error&quot;: {&quot;code&quot;: 401, &quot;message&quot;: &quot;Invalid&quot;, &quot;context&quot;: &quot;auth&quot;}} </code></pre><ul> <li><p><code>&quot;code&quot;</code> may have one of the following values:</p> <p> <code>-1</code> means the request message is not a proper JSON string</p> <p> <code>400</code> means the request is not properly formed</p> <p> <code>401</code> means the credentials in <code>&quot;auth&quot;</code> are invalid</p> <p> <code>500</code> means an internal error occured while generating the response message. Individual calls may or may not have completed successfully.</p> <p> <code>501</code> means the application of the given arguments to the specified procedure is not supported.</p> </li> </ul> <ul> <li><p><code>&quot;context&quot;</code> indicates where the failure happened. If present, it may have one of the following values:</p> <p> <code>&quot;arguments&quot;</code> if arguments for a procedure are missing</p> <p> <code>&quot;auth&quot;</code> if the auth field is of invalid format</p> <p> <code>&quot;calls&quot;</code> if the calls field is of invalid format, i.e. not a list</p> <p> <code>&quot;procedure&quot;</code> if the procedure is missing</p> </li> </ul> <h1 id="time-series-data">Time Series Data</h1> <h2 id="read">read</h2> <p>Read data from a resource.</p> <pre><code>{ &quot;procedure&quot;: &quot;read&quot;, &quot;arguments&quot;: [ &lt;ResourceID&gt;, { &quot;starttime&quot;: number = 0 &quot;endtime&quot;: number = &lt;current unix time&gt;, &quot;sort&quot;: &quot;asc&quot; | &quot;desc&quot; = &quot;desc&quot; &quot;limit&quot;: number = 1, &quot;selection&quot;: &quot;all&quot; | &quot;givenwindow&quot; | &quot;autowindow&quot; = &quot;all&quot;, } ], &quot;id&quot;: 1 } </code></pre><ul> <li><code>&lt;ResourceID&gt;</code> is the identifier of the device to read. See <a href="#identifying-resources">Identifying Resources</a> for details.</li> <li><code>&quot;starttime&quot;</code> and <code>&quot;endtime&quot;</code> are <a href="http://en.wikipedia.org/wiki/Unix_time">Unix timestamps</a> that specify the window of time to read. <code>&quot;starttime&quot;</code> defaults to <code>0</code> and <code>&quot;endtime&quot;</code> defaults to the current time.</li> <li><code>&quot;sort&quot;</code> defines the order in which points should be ordered: ascending (<code>&quot;asc&quot;</code>) or descending (<code>&quot;desc&quot;</code>) timestamp order. Defaults to <code>&quot;desc&quot;</code>.</li> <li><code>&quot;limit&quot;</code> sets a maximum on the number of points to return. <code>&quot;limit&quot;</code> is applied after the results have been sorted, so different values of <code>&quot;sort&quot;</code> will return different sets of points. Defaults to <code>1</code>.</li> <li><code>&quot;selection&quot;</code> supports downsampling. <code>&quot;givenwindow&quot;</code> splits the time window evenly into <code>&quot;limit&quot;</code> parts and returns at most one point from each part. <code>&quot;autowindow&quot;</code> samples evenly across points in the time window up to <code>&quot;limit&quot;</code>. Note that these options provide a blind sampling function, not averaging or other type of rollup calculation. Defaults to <code>&quot;all&quot;</code>, which turns off downsampling, returning all points up to <code>&quot;limit&quot;</code>.</li> </ul> <h4 id="response">response</h4> <p>Response is a list of <a href="http://en.wikipedia.org/wiki/Unix_time">timestamp</a>, value pairs.</p> <pre><code> // float resource { &quot;status&quot;: &quot;ok&quot;, &quot;result&quot;: [[1376709527,64.2]], &quot;id&quot;: 1 } // string resource { &quot;status&quot;: &quot;ok&quot;, &quot;result&quot;: [[1376950234,&quot;World&quot;],[1376950230,&quot;Hello&quot;]], &quot;id&quot;: 1 } // integer resource { &quot;status&quot;: &quot;ok&quot;, &quot;result&quot;: [[1376950410,11],[1376950405,10]], &quot;id&quot;: 1 } </code></pre><h4 id="example">Example</h4> <p>Read the most recent single value for a datasource:</p> <pre><code>$ curl -d &#39;{&quot;auth&quot;:{&quot;cik&quot;:&quot;bef3c7f91ac3562e6a2212345678901234567890&quot;},&quot;calls&quot;:[{&quot;procedure&quot;:&quot;read&quot;,&quot;arguments&quot;:[{&quot;alias&quot;: &quot;log&quot;}, {}],&quot;id&quot;:0}]}&#39; -H &#39;Content-type: application/json; charset=utf-8&#39; http://m2.exosite.com/onep:v1/rpc/process [{&quot;id&quot;:0,&quot;status&quot;:&quot;ok&quot;,&quot;result&quot;:[[1390622248,&quot;test value&quot;]]}] </code></pre><p>Read the earliest two points for a datasource:</p> <pre><code>$ curl -d &#39;{&quot;auth&quot;:{&quot;cik&quot;:&quot;bef3c7f91ac3562e6a2212345678901234567890&quot;},&quot;calls&quot;:[{&quot;procedure&quot;:&quot;read&quot;,&quot;arguments&quot;:[{&quot;alias&quot;: &quot;log&quot;}, {&quot;sort&quot;:&quot;asc&quot;, &quot;limit&quot;:2}],&quot;id&quot;:0}]}&#39; -H &#39;Content-type: application/json; charset=utf-8&#39; http://m2.exosite.com/onep:v1/rpc/process [{&quot;id&quot;:0,&quot;status&quot;:&quot;ok&quot;,&quot;result&quot;:[[1390622242,&quot;second value&quot;],[1390622240,&quot;first value&quot;]]}] </code></pre><h2 id="write">write</h2> <p>Writes a single value to the resource specified.</p> <pre><code>{ &quot;procedure&quot;: &quot;write&quot;, &quot;arguments&quot;: [ &lt;ResourceID&gt;, &lt;value&gt; ], &quot;id&quot;: 1 } </code></pre><ul> <li><p><code>&lt;ResourceID&gt;</code> is the identifier of the device to write. See <a href="#identifying-resources">Identifying Resources</a> for details.</p> </li> <li><p><code>&lt;value&gt;</code> is the value to write.</p> </li> </ul> <h4 id="response">response</h4> <pre><code>{ &quot;status&quot;: &quot;ok&quot;, &quot;id&quot;: 1 } </code></pre><hr> <h2 id="writegroup">writegroup</h2> <p>Writes the given values for the respective resources in the list. The difference between calling <code>writegroup</code> and combining multiple <code>write</code> calls into a single request is that <code>writegroup</code> guarantees the points are written at the same timestamp.</p> <pre><code>{ &quot;procedure&quot;: &quot;writegroup&quot;, &quot;arguments&quot;: [ [[&lt;ResourceID&gt;, &lt;value&gt;], ...] ], &quot;id&quot;: 1 } </code></pre><ul> <li><p><code>&lt;ResourceID&gt;</code> is the identifier of the resource to which to write. See <a href="#identifying-resources">Identifying Resources</a> for details.</p> </li> <li><p><code>&lt;value&gt;</code> is the value to write.</p> </li> </ul> <h4 id="response">response</h4> <pre><code>{ &quot;status&quot;: &quot;ok&quot;, &quot;id&quot;: 1 } </code></pre><hr> <h2 id="record">record</h2> <p>Records a list of historical entries to the resource specified. NOTE: this API is deprecated. Please use <a href="#recordbatch"><code>recordbatch</code></a> instead.</p> <pre><code>{ &quot;procedure&quot;: &quot;record&quot;, &quot;arguments&quot;: [ &lt;ResourceID&gt;, [[&lt;timestamp&gt;, &lt;value&gt;], ...], {} ], &quot;id&quot;: 1 } </code></pre><ul> <li><code>&lt;ResourceID&gt;</code> is a resource identifier. See <a href="#identifying-resources">Identifying Resources</a> for details.</li> <li><p>The second argument is a list of timestamp, value entries to record to the resource. If <code>&lt;timestamp&gt;</code> is a negative value, it means an offset back into the past from the current time.</p> </li> <li><p>The behavior of the platform when <code>&lt;timestamp&gt;</code> is set to times in the future is undefined.</p> </li> <li>The third argument is currently unused.</li> </ul> <h4 id="response">response</h4> <pre><code>{ &quot;status&quot;: &quot;ok&quot;, &quot;id&quot;: 1 } </code></pre><ul> <li><code>&quot;status&quot;: &quot;ok&quot;</code> means the entries were successfully recorded.</li> </ul> <hr> <h2 id="recordbatch">recordbatch</h2> <p>Records a list of historical entries to the resource specified. </p> <pre><code>{ &quot;procedure&quot;: &quot;recordbatch&quot;, &quot;arguments&quot;: [ &lt;ResourceID&gt;, [[&lt;timestamp&gt;, &lt;value&gt;], ...] ], &quot;id&quot;: 1 } </code></pre><ul> <li><p><code>&lt;ResourceID&gt;</code> is a resource identifier. See <a href="#identifying-resources">Identifying Resources</a> for details.</p> </li> <li><p>The second argument is a list of timestamp, value entries to record to the resource. If <code>&lt;timestamp&gt;</code> is a negative value, it means an offset back into the past from the current time.</p> </li> <li><p>The behavior of the platform when <code>&lt;timestamp&gt;</code> is set to times in the future is undefined.</p> </li> </ul> <h4 id="response">response</h4> <pre><code>{ &quot;status&quot;: &quot;ok&quot;, &quot;id&quot;: 1 } </code></pre><ul> <li><code>&quot;status&quot;: &quot;ok&quot;</code> means the entries were successfully recorded.</li> </ul> <pre><code>{ &quot;status&quot;: [[&lt;timestamp&gt;, &#39;invalid&#39;], ...], &quot;id&quot;: 1 } </code></pre><ul> <li>If <code>&quot;status&quot;</code> is not <code>&quot;ok&quot;</code>, it will return an array to indicate which <code>&lt;timestamp&gt;</code> is failed to be writen.</li> </ul> <h4 id="example">example</h4> <pre><code>{ &quot;arguments&quot;: [ &quot;553d1367fa4613f0eea894d1677a13ef1f091150&quot;, [[1421906172,30],[1421906160,20]] ], &quot;id&quot;: 1, &quot;procedure&quot;: &quot;recordbatch&quot; } response: { &quot;status&quot;: [[1421906172, &#39;invalid&#39;]], &quot;id&quot;: 1 } </code></pre><hr> <h2 id="flush">flush</h2> <p>Empties the specified resource of data per specified constraints. If no constraints are specified, all data gets flushed. </p> <pre><code>{ &quot;procedure&quot;: &quot;flush&quot;, &quot;arguments&quot;: [ &lt;ResourceID&gt;, { &quot;newerthan&quot;: number, &quot;olderthan&quot;: number } ], &quot;id&quot;: 1 } </code></pre><ul> <li><code>&lt;ResourceID&gt;</code> specifies what resource to flush. See <a href="#identifying-resources">Identifying Resources</a> for details.</li> </ul> <ul> <li><code>&quot;newerthan&quot;</code> and <code>&quot;olderthan&quot;</code> are optional timestamps that constrain what data is flushed.
